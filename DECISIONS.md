# 決策紀錄 — GTC Taipei 2026 多受眾導覽地圖

> 記錄這個專案做了哪些關鍵決策、考慮過的選項、為什麼這樣選。方便日後回顧與調整。
> 最後更新：2026-06-02

## 1. 最終形式：互動式網頁
- **選項**：互動式網頁 / Markdown 知識庫 / 資料表(CSV) / 先做結構模板
- **選擇**：互動式網頁（單一 HTML）
- **理由**：可直接分享、可上線、體驗最好；能內嵌影片並做互動。

## 2. 內容來源：官方逐字稿為準
- **演進**：一開始想用 WebFetch 抓 YouTube 資訊 → 抓得到媒體整理但抓不到逐字稿/時間戳 → 你提供官方 SRT（`transcript.srt`）。
- **選擇**：以官方逐字稿為**權威來源**，媒體報導僅作脈絡補充。
- **理由**：內容能對齊黃仁勳實際發言、避免媒體轉述失真；每個主題附「黃仁勳原句」。

## 3. 時間戳：真實時間（非估計）
- **演進**：抓不到時間戳時先用「估算時間段」並標註估計 → 拿到逐字稿後全部換成真實時間。
- **選擇**：13 主題全部採逐字稿真實時間戳。
- **理由**：點主題能精準跳到該片段。對照表見 `keynote-timestamps.md`。

## 4. 介面形態：可拖拉畫布心智圖（非捲動頁）
- **演進**：原規劃捲動式頁面（受眾切換 + 主題卡 + 矩陣）→ 你希望改成可拖拉畫布 + 從核心向外展開的心智圖。
- **選擇**：三層心智圖 — 核心選視角 → 放射展開 13 主題 → 點主題開詳情+影片。
- **理由**：更貼近「多受眾導覽」的探索感。

## 5. 主題數與結構：9 → 13
- **演進**：研究階段先用推測的 9 主題 → 逐字稿揭露漏掉的大段落（Vera CPU、企業 AI 工具組/Nemotron 3 Ultra/Cadence、CUDA-X、收尾）。
- **選擇**：重整為 **13 個主題**，對齊演講真實結構與時間軸。
- **理由**：完整覆蓋整場 keynote、不遺漏關鍵段落。

## 6. 畫布技術：React Flow (@xyflow)
- **選項**：React Flow / 輕量 vanilla + Panzoom 小套件 / markmap
- **選擇**：React Flow（業界標準節點畫布，MIT）
- **理由**：拖拉/縮放/minimap/控制列/連線內建；節點是自訂 component，可塞受眾卡片與影片。純 canvas 繪圖庫（Cytoscape/vis.js/Sigma）不適合，因為節點塞不了 iframe。

## 7. 其他開源美化
- **採用**：Google Fonts（Noto Sans TC + Space Grotesk）、Lucide 圖示、React Flow 內建 Background/Controls/MiniMap。
- **保留手作**：節點環狀/聚焦佈局（三角函數，省一包 CDN）、詳情面板版型（避免框架味）。
- **理由**：用開源處理通用、易出錯的部分；設計本身保留掌控。

## 8. 打包方式：單一 HTML、免 build
- **選擇**：ESM CDN（esm.sh）+ importmap + htm 取代 JSX，零 build step。
- **細節**：`@xyflow/react@12?external=react,react-dom` 與 React 共用同一份 React；補上 `react/jsx-runtime` 對應（否則整頁不渲染）。
- **授權**：保留 React Flow 預設 attribution（隱藏需付費方案）。
- **理由**：維持「單一檔、可雙擊/可分享」，同時用現成函式庫。

## 9. 影片播放：每主題獨立 iframe
- **演進**：原本用單一共用 YouTube Player → 切換主題幾次後變黑（舊 player 被卸載但全域變數還指向它）。
- **選擇**：每個主題一個獨立 `<iframe>`（`key={主題id}` + `?start=秒數`）。
- **理由**：React 每次渲染全新 iframe，徹底消除共用播放器的生命週期 bug。

## 10. file:// 的影片限制與 fallback
- **現象**：直接雙擊（`file://`）時影片內嵌被擋（Error 153）。
- **根因**：承載頁面 `file://` 的 origin 是 `null`，YouTube 內嵌握手失敗（與 YouTube 是否 https 無關）。
- **選擇**：偵測 `file://` 時，影片區改顯示「在 YouTube 開啟這一段（帶時間戳）」按鈕；用 http server 開則正常內嵌。
- **理由**：兩種開啟方式都有合理體驗。詳見 `README.md`。

## 11. 受眾重點凸顯：點亮核心 + 其餘變暗
- **選項**：點亮核心+其餘變暗 / 只加「重點」標記不變暗 / 三層（核心/相關/次要）
- **選擇**：兩層 — 核心主題點亮（光暈 + ⭐重點徽章 + 粗連線），其餘淡化但仍可點。
- **理由**：聚焦感最強、一眼看出該受眾該看哪些；資訊不丟失。

## 12. 各受眾的「核心主題」對照（可調）

| 受眾 | 核心主題 |

| 知識工作者 | Agentic AI、CUDA-X、企業 AI 工具組、RTX Spark |

## 13. 全貌 / 聚焦 可切換（預設全貌）
- **動機**：有人想看大局、有人想快速看重點。
- **選擇**：header 加「全貌 / 聚焦重點」切換，預設全貌。
  - 全貌：13 主題固定環狀，核心仍點亮。
  - 聚焦：核心聚到該視角旁、非核心退外圈並隱藏連線。
- **理由**：尊重不同使用意圖，一鍵切換。

## 14. 製作流程：多代理工作流
- **選擇**：用 Workflow 編排 — 13 個研究 agent（各自以黃仁勳原句為依據 + 媒體脈絡，產出四受眾視角）並行，再由 1 個 build agent 組裝 HTML。
- **理由**：平行加速、內容有事實依據、時間戳由腳本固定不讓 agent 亂猜。

## 15. 版本控管
- **選擇**：`git init` 並提交（純本機 repo，無遠端）。`.claude/` 已排除。

## 16. 投資者視角加「關聯公司 / 成長與未來性 / 參考資料」
- **動機**：投資者真正在意的是「NVIDIA 產品最後關聯到哪些公司」與「每個產品的成長性／未來性」。
- **選擇**：每個主題的投資者視角多三塊 —— 關聯公司（依分類上色 chips、hover 看角色說明）、成長與未來性（NVIDIA 演講說法，附「非投資建議」聲明）、參考資料（真實連結）。
- **來源**：由聚焦工作流產生；**agent 先讀 `REFERENCES.md` 挑已查證連結**，不足再補搜（避免亂搜、來源一致）。
- **資料結構**：`AUDIENCE_EXTRA = { investor: { <themeId>: { companies[{name,tag,note}], growth, sources[{title,url}] } } }`，**刻意設計成可擴充** —— 之後要替知識工作者／一般消費者／公司決策者各加專屬維度，往這個物件加 key 即可。
- **不重跑 build**：以程式注入既有 HTML（保留先前手動修正：jsx-runtime、每主題 iframe、file:// fallback、核心點亮、全貌/聚焦），未重跑整個 build agent。

## 17. 影片全螢幕修正
- **問題**：影片 iframe 全螢幕鈕無作用。
- **原因**：`allow=` 清單漏了 `fullscreen`，且 React/htm 需用 camelCase `allowFullScreen` 屬性才會輸出。
- **選擇**：`allow` 補 `fullscreen` + 改用 `allowFullScreen`。

## 18. 關聯公司改為「按需展開的下一階心智圖節點」（drawer 瘦身）
- **動機**：把關聯公司列在 drawer 會讓資料量太多太雜；且改成節點更符合心智圖往下長的隱喻。
- **考量**：全部一次節點化會爆炸（13×~10≈130 個）且跨主題重複，故採「按需、單一主題」展開。
- **選擇**：drawer 內關聯公司改為一顆按鈕「在地圖上展開關聯公司（N 家）」；點了才於該主題節點旁**扇形長出公司子節點**（依分類上色、邊以分類色連接、可收合）。點公司節點回到該主題詳情；切換受眾/重置會清空。drawer 只留成長與未來性、參考資料、聲明，保持精簡。
- **狀態**：四層心智圖（核心→視角→主題→關聯公司），目前公司節點僅投資者有資料。

## 19. 公司節點用「球泡」，大小 = 對 NVIDIA 曝險（不是市值）
- **取捨**：球 vs 卡片 → 選球。因為要把「曝險」編碼成「大小」，大小編碼只在簡單形狀（圓）上讀得清楚；10 張含說明的卡片會變文字牆，球能一眼看出「誰跟 NVIDIA 綁最深」。
- **為何不用市值**：大公司 ≠ 跟 NVIDIA 關係深（微軟市值大但曝險中等；純受惠台廠曝險高）。對投資者，曝險比市值有意義。市值還有未上市無值、量級差距極大、會過期等問題。
- **編碼**：球大小 = 曝險（high/medium/low 三級）、顏色 = 類別、名稱在球下。
- **補回卡片優點**：點球泡彈出該公司「角色／為何受惠 + 曝險分級」小卡（hover 也有），手機也看得到。
- **曝險資料**：由單一 agent 一次評定全部 127 家（確保跨主題校準一致），分布 高 61／中 53／低 13；存於 `AUDIENCE_EXTRA…companies[].exposure`。
- **可調**：三級對應的球直徑在 CSS `.exp-high/.exp-med/.exp-low`；評級可改 `exposure` 欄位。
- **用詞**：畫面標籤用白話「與 NVIDIA 關聯深度」（高/中/低），不用行話「曝險」——「險」字易誤讀為危險、且競爭對手用「關聯淺」比「受惠低」通順；僅在圖例附一個小括號「（投資上的曝險）」給財經讀者。

## 20. 投資者核心改用「產品＋論述」鏡頭（補上 PC 線與 Physical AI）
- **原本的選法**：用「投資論述」鏡頭（護城河、資本支出、營收模型、旗艦量產、供應鏈框架）→ 偏抽象，漏了 PC/機器人等新產品線。
- **修正後的鏡頭**：對這個工具，公司/成長本來就掛在「產品」上，所以核心改跟著「開了明確營收線/新 TAM、有清楚受惠鏈」的產品＋關鍵論述走。
- **新投資者核心（6）**：Vera Rubin 量產、AI 工廠/DSX、Token 經濟學、**N1X/新 PC 線**、**Physical AI（Cosmos）**、CUDA-X；開場/台灣生態系降為非核心。
- **理由**：N1X = 全新 PC TAM + 聯發科/台積電；Physical AI = 機器人/自駕長線新市場——都是投資者該看的成長引擎。

## 21. 公司只接「產品」主題；概念/論述主題改指引（＋展開時自動聚焦）
- **問題**：概念/論述主題（開場、Agentic、Token 經濟學、收尾）也接公司很怪——概念沒有供應鏈，其「公司」只是論述成立時的間接受惠、且與產品主題重複。
- **心智模型**：**公司 = 產品的供應鏈**。
- **選擇**：`CONCEPT_THEMES` 內的主題不顯示「展開公司球泡」，改一行指引「受惠公司見各產品主題」；產品主題維持球泡。成長/參考資料對概念主題仍保留。
- **自動聚焦**：展開某主題的公司球泡時，其他 12 個主題大幅淡化（`.node-theme.faded`），只留當前主題＋其球泡，減少「主題線＋球泡」混雜。

## 22. 互動改「畫布優先」，收斂來回跳轉
- **問題**：展開球泡的鈕埋在右側 drawer，使用者注意力在 節點→右側 drawer→畫布球泡→底部小卡→drawer 之間來回跳，很不順手。
- **選擇（畫布優先）**：
  - 點**產品主題節點本體** = 就地展開／收合公司球泡（投資者；非概念）。
  - 節點上加「**詳情**」小鈕 → 才開 drawer 看影片/成長/來源（選配）。
  - **概念主題 / 非投資者** → 點節點仍直接開 drawer（沒有球泡可展）。
  - 公司角色小卡改為**就近彈出**（以點擊座標定位、夾在視窗內），不再跑到底部。
- **效果**：探索動作幾乎都留在畫布、drawer 變選配，減少區域切換。

## 23. 供應鏈名單查核 + 3 色簡化 + 標註依據
- **查核**：用工作流（每主題一 agent，比對 `transcript.srt` + web）逐家判定，移除 4 個錯誤（如「鎧俠／Cadence」兩家被錯黏、張冠李戴的 MSI／聯發科、過度歸因的台達電），修正多項名稱、補上漏列（CoreWeave/Nebius/Dassault/Adobe…）。
- **標註依據（provenance）**：每家加 `basis` —— keynote 點名 / 已知供應鏈·夥伴 / 合理推測 / 競爭對手（點名 58、供應鏈 54、推測 15、競爭 9）。角色小卡顯示「依據：…」，讓「名單正不正確」透明。
- **3 色簡化**：原本 8 種類別顏色太雜。改用「身分」三色 —— **台廠（綠，台股可投資）／海外夥伴（灰，脈絡）／競爭對手（紅）**；細類別改放角色卡文字。球大小仍 = 與 NVIDIA 關聯深度。
- **可調**：`ORIGIN_COLOR` 三色；台廠判定在注入腳本的 TW 關鍵字清單；`basis`/`origin` 存於 `AUDIENCE_EXTRA…companies[]`。
- **手動修正**：NVIDIA 本尊不列為關聯公司（它是主體，已全數移除）；GMI Cloud 非台廠（美國 GPU 雲新創，origin 改 overseas）。

## 24. 受眾重新校準（對齊內容與 LinkedIn 通路）
- **問題**：四類「概念對稱、實際不對稱」——投資者最實、消費者最虛（GTC 內容對一般消費者多為間接）。
- **校準**：知識工作者→**開發者·技術工作者**、一般消費者→**科技愛好者**（LinkedIn 最廣的一群，內容改「大局/意義」而非「你會買什麼」）、公司決策者→**企業決策者**；投資者明確定位為**散戶「該買哪些股票」**。開發者與科技愛好者內容全面重寫、重設核心主題。

## 25. 投資者（散戶）：上市狀態＋股票代號
- **動機**：散戶要「買得到」。每家關聯公司標 listed/ticker/exchange。
- **未上市**（GMI Cloud、Cursor、Nscale、Anthropic/OpenAI、Unitree、林科、Red Pill 等）明確標「未上市」、球泡改空心虛線；角色卡顯示代號與備註。保留非投資建議聲明。
- **作法**：工作流查代號（87 家）＋正規化比對＋手動補漏（Arm、Dell、現代、聯想、宏碁、高通、ServiceNow、Nebius、Siemens 等）。

## 26. 部署：GitHub Pages
- **選擇**：主檔改名 `index.html`，`gh` 建立公開 repo `brennenhuang/gtc-2026-keynote-map`，開啟 Pages（main 分支根目錄）。
- **網址**：https://brennenhuang.github.io/gtc-2026-keynote-map/ —— https origin 讓影片可正常內嵌；之後 `git push` 自動更新。

## 27. 台灣個股 → 相關受惠族群（只列族群、不列個股）
- **動機**：散戶買台積電，想知道「旁邊還有哪些族群跟著吃單」（載板、封測、CoWoS、散熱…）。
- **作法**：工作流對 14 支台股 anchor 各查相關受惠族群（含代表股，但**依使用者要求只在 UI 顯示族群+理由，不列個股**；底層資料也不存個股，避免像報明牌）。
- **呈現**：點台灣已上市個股球泡 → 角色小卡新增「相關受惠族群」清單（族群名＋一句受惠理由），小卡可捲動。
- **免責**：族群區塊底部明標「不構成投資建議，投資請自行評估風險」。
- **資料**：`BENEFICIARIES[代號] = { name, groups:[{group, why}] }`；popover 以個股主代號查詢。
